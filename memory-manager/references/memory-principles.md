# 记忆蒸馏原则与反模式

## 核心原则

### 原则一：具体胜于抽象

记忆的价值在于下次能直接用，而不是"知道有这件事"。

| 🔴 坏的记忆 | 🟢 好的记忆 |
|---|---|
| "部署有一些特殊要求" | "部署前必须先跑 `pnpm test:e2e`，否则 Vercel 成功部署但生产测试会失败" |
| "用户不喜欢某些代码风格" | "禁用 class，偏好纯函数；未经询问不得引入新依赖" |
| "第三章有问题" | "第三章结尾视角跳跃，需从女主视角切回，现停在"然而三天后……"这句话前" |
| "研究方向要注意" | "受访者 B 对匿名性敏感，引用时必须模糊职位描述，不能出现具体部门名" |
| "数据库操作要小心" | "不能直接用 `db.query()`，必须通过 `lib/db.ts` 包装器（含连接池和错误重试）" |

---

### 原则二：蒸馏不是搬运

**错误做法**：把对话内容直接粘贴进记忆文件。
**正确做法**：提炼出"下次执行这个任务需要知道什么"。

```
原始事件 → 去掉时间/情境细节 → 保留可操作的规则 → 写入记忆
```

**编程示例**：
```
原始事件：今天把 MySQL 迁移到 PostgreSQL，发现 JSON 字段语法不兼容，折腾了两小时。

❌ 搬运："2026-04-04 完成了数据库迁移工作"
✓ 蒸馏："MySQL→PostgreSQL：JSON_EXTRACT() 语法不兼容，迁移前必须全局替换"
```

**写作示例**：
```
原始事件：今天讨论了第二章节奏，用户觉得对话太密集，需要穿插更多环境描写。

❌ 搬运："2026-04-04 讨论了第二章修改方向"
✓ 蒸馏："第二章节奏：每 3 段对话后需插入至少 1 段环境/心理描写"
```

**研究示例**：
```
原始事件：整理文献时发现 Smith(2019) 和 Lee(2022) 结论直接矛盾，需在论文中处理。

❌ 搬运："2026-04-04 阅读了 Smith 和 Lee 的论文"
✓ 蒸馏："Smith(2019) vs Lee(2022) 结论相悖——前者支持正相关，后者否定——第三章须正面处理此分歧"
```

---

### 原则三：容量边界倒逼质量

`hot.md` 的 600 tokens 上限不是限制，是**保证信息质量的机制**。

当热记忆满了，必须做选择：
- 几乎每次都有用 → 留在 `hot.md`
- 偶尔有用但不是每次 → 下沉到 `context.md`
- 其实很少用到了 → 删除

---

### 原则四：人与工作分离

**`context.md`** 描述的是"这个工作/项目"——换了工作就换 context.md。
**`user.md`** 描述的是"这个人"——跨所有工作通用，可全局复用。

判断方法：这条信息换个项目/领域还适用吗？适用 → `user.md`，不适用 → `context.md`。

---

## 常见反模式

### 反模式一：热记忆膨胀

**症状**：`hot.md` 越来越长，超过 600 tokens，什么都有但什么都找不到。
**根因**：只加不减，没有定期审计。
**修复**：每月审计一次，将 30 天未用到的条目下沉到 `context.md` 或直接删除。

---

### 反模式二：过时记忆

**症状**：`hot.md` 还写着"用 yarn"，但已切换到 pnpm 三个月；或写着"第三章未完成"，但早就交稿了。
**根因**：记忆只增不改，没有更新机制。
**修复**：发现记忆与现实不符时立即更新，不要等到"下次专门整理"。

---

### 反模式三：假分层

**症状**：`hot.md` 写着"详见 context.md"，但 `context.md` 只是把 `hot.md` 的内容重复了一遍。
**根因**：建立了分层结构，但没有真正分层。
**修复**：`hot.md` 只放"摘要+指针"，`context.md` 放"完整细节"。两者互补，不重复。

---

### 反模式四：记忆代替思考

**症状**：记忆里塞满了通用知识（"如何用 TypeScript"、"写作的基本原则"）或大段文档摘录。
**根因**：把记忆当知识库，而不是"这个工作的特殊补充"。
**修复**：只记录**不能从文档/训练中获得**的信息：特有规则、踩过的坑、用户偏好。

---

### 反模式五：SOP 遗漏

**症状**：同样的多步骤操作每次都要重新解释，或每次都出现同样的错误。
**根因**：复杂流程没有文档化，靠"临时发现"。
**修复**：第一次完整执行一个新的复杂流程后，立即创建 SOP 文件，包含所有容易出错的步骤和验证方法。

---

## 层级选择指南

```
这条信息：

几乎每次工作都会用到？
  YES → hot.md（≤ 一两句话）
  NO  ↓

偶尔需要，但忘了会出错？
  YES → context.md（详细记录）
  NO  ↓

是可重复执行的多步骤流程？
  YES → sop/[task].md（SOP 格式）
  NO  ↓

是关于我这个人的偏好/风格？
  YES → user.md
  NO  ↓

可以从文档/训练/搜索中直接查到？
  YES → 不需要记，删除
  NO  → 再想想是否真的有价值
```

## 优质记忆示例

### 编程项目（context.md）

```markdown
## 已知雷区

- `docker build` 必须从项目根目录执行——.dockerignore 路径相对根目录，子目录执行会把 node_modules 打进镜像
- E2E 测试前必须运行 `pnpm seed:test`，否则因缺少测试数据失败
- 单元测试和 E2E 不能并行跑（端口冲突），用 `pnpm test`，不加 `--parallel`
```

### 写作项目（context.md）

```markdown
## 已知约定

- 全文视角锁定女主，不得切入其他角色内心
- 第三章节奏：每 3 段对话后插入至少 1 段环境/心理描写
- 人物"陈默"口头禅"也许吧"，每章出现不超过 2 次

## 遗留问题

- 第五章结尾悬念未解（读者知道信的内容，女主不知道），第六章需处理信息差
```

### 研究项目（context.md）

```markdown
## 访谈限制

- 受访者 B：高度敏感，引用时必须模糊职位，不出现具体部门名
- 受访者 D：仅授权匿名引用，不得关联任何可识别信息

## 文献分歧待处理

- Smith(2019) vs Lee(2022)：结论相悖，第三章须正面处理，不能回避
```

### 用户画像（user.md，跨领域通用）

```markdown
## 协作风格

- 重要改动先给 plan，等确认后执行；影响小的直接做，完成后说明
- 遇到不确定的地方直接问，不猜，不静默假设

## 表达偏好

- 解释简洁，先结论后细节
- 方向有问题直接说，不要迂回铺垫
- 不喜欢过度的肯定语气（"很好！""当然！"）

## 禁忌

- 未经询问不得引入新工具/依赖/框架
- 不做任何"先做了再说"的不可逆操作
```
